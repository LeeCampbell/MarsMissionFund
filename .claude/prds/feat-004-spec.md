# PRD: feat-004 — Account Registration & Onboarding

> Implementation-ready specification.
> Generated by Spec Writer.
> Feature brief: feat-004-account-registration-onboarding.md
> Research: feat-004-research.md

## Overview

Implement the account onboarding lifecycle from first authenticated request through role selection, profile setup, notification preferences, and onboarding completion.
This feature replaces the existing onboarding placeholder with a multi-step wizard, adds profile and preference management endpoints, extends the Account domain entity with mutation methods, and introduces the first domain event emission to the event store.
The account record established here is the identity foundation that all other bounded contexts (Campaign, Donor, Payments) reference.

## User Stories & Acceptance Criteria

### US-001: Onboarding Welcome

**As a** newly registered user
**I want to** see a welcome screen when I first log in
**So that** I understand what Mars Mission Fund is about before setting up my account.

**Acceptance Criteria:**

- **Given** a user with `onboarding_completed = false` and `onboarding_step = 'welcome'` **When** they navigate to `/onboarding` **Then** the welcome step is displayed with a brand-appropriate welcome message and a primary CTA to begin onboarding.
- **Given** a user on the welcome step **When** they click the primary CTA **Then** the `onboarding_step` is updated to `role_selection` via `PATCH /api/v1/accounts/me/onboarding` and the role selection step is displayed.

### US-002: Role Selection

**As a** new user
**I want to** choose whether I am a backer, creator, or both
**So that** the platform can tailor my experience to my intent.

**Acceptance Criteria:**

- **Given** a user on the `role_selection` step **When** the step renders **Then** three options are displayed: "Backer" (default, pre-selected), "Creator", and "Both".
- **Given** a user who selects "Creator" or "Both" **When** the selection is displayed **Then** an informational message appears stating that KYC verification will be required to access creator features (no blocking action, just information).
- **Given** a user who selects "Creator" **When** they confirm the step **Then** `roles` is updated to `['backer', 'creator']` (creator is additive to backer, never replaces it).
- **Given** a user who selects "Both" **When** they confirm the step **Then** `roles` is updated to `['backer', 'creator']`.
- **Given** a user who selects "Backer" (default) **When** they confirm the step **Then** `roles` remains `['backer']`.
- **Given** a user on the role selection step **When** they confirm their selection **Then** the `onboarding_step` advances to `profile` and the profile step is displayed.
- **Given** a user on the role selection step **When** they click a "Back" control **Then** they return to the welcome step (client-side navigation only, `onboarding_step` is NOT regressed server-side).

### US-003: Profile Setup

**As a** new user
**I want to** set my display name, bio, and avatar URL during onboarding
**So that** other users can identify me on the platform.

**Acceptance Criteria:**

- **Given** a user on the `profile` step **When** the step renders **Then** form fields are shown for display name (text input), bio (textarea), and avatar URL (text input). All fields are optional.
- **Given** a user who enters a display name **When** the input is validated **Then** it must be 1-100 characters after trimming, or empty (skipped). Whitespace-only strings are rejected.
- **Given** a user who enters a bio **When** the input is validated **Then** it must be 0-500 characters after trimming.
- **Given** a user who enters an avatar URL **When** the input is validated **Then** it must be a valid URL (https scheme) or empty.
- **Given** a user on the profile step **When** they click "Continue" **Then** valid fields are persisted via `PATCH /api/v1/accounts/me`, `onboarding_step` advances to `preferences`, and the preferences step is displayed.
- **Given** a user on the profile step **When** they click "Skip" **Then** no profile data is saved, `onboarding_step` advances to `preferences`, and the preferences step is displayed.
- **Given** a user on the profile step **When** they click a "Back" control **Then** they return to the role selection step (client-side only, no server regression).

### US-004: Notification Preferences

**As a** new user
**I want to** configure which notifications I receive
**So that** I only get communications I care about.

**Acceptance Criteria:**

- **Given** a user on the `preferences` step **When** the step renders **Then** toggle switches are shown for each notification category with the following defaults:
  - Campaign updates: ON
  - Milestone completions: ON
  - Contribution confirmations: ON
  - New campaign recommendations: ON
  - Platform announcements: OFF
  - Security alerts: ON (displayed as always-on, toggle is disabled/non-interactive)
- **Given** a user who toggles a notification preference **When** the toggle changes **Then** the change is reflected in local state (not persisted until the step is confirmed).
- **Given** a user who attempts to toggle security alerts **When** they interact with the security alerts toggle **Then** nothing happens; the toggle remains ON with a tooltip or helper text: "Security alerts cannot be disabled."
- **Given** a user on the preferences step **When** they click "Complete" **Then** preferences are persisted via `PATCH /api/v1/accounts/me/preferences`, `onboarding_step` is set to `completed`, `onboarding_completed` is set to `true`, and the completion step is displayed.
- **Given** a user on the preferences step **When** they click "Skip" **Then** default preferences are kept (not explicitly saved, they are the DB defaults), `onboarding_step` is set to `completed`, `onboarding_completed` is set to `true`, and the completion step is displayed.

### US-005: Onboarding Completion

**As a** user who has completed onboarding
**I want to** see a confirmation screen and proceed to the dashboard
**So that** I know my setup is done and I can start using the platform.

**Acceptance Criteria:**

- **Given** a user who has just completed onboarding **When** the completion step renders **Then** a congratulatory message is displayed with a personalised greeting (using display name if set, or a fallback) and a primary CTA "Go to Dashboard".
- **Given** a user on the completion step **When** they click "Go to Dashboard" **Then** they are navigated to `/dashboard`.
- **Given** a user who has already completed onboarding **When** they navigate to `/onboarding` directly **Then** they are redirected to `/dashboard`.

### US-006: Resume Onboarding

**As a** user who abandoned onboarding
**I want to** resume from where I left off
**So that** I do not have to repeat completed steps.

**Acceptance Criteria:**

- **Given** a user with `onboarding_completed = false` and `onboarding_step = 'profile'` **When** they log in and are redirected to `/onboarding` **Then** the wizard starts at the profile step.
- **Given** a user who refreshes the browser during onboarding **When** the page reloads **Then** the wizard resumes from the server-persisted `onboarding_step`.

### US-007: Profile Management (Post-Onboarding)

**As a** registered user
**I want to** edit my profile and notification preferences after onboarding
**So that** I can keep my information up to date.

**Acceptance Criteria:**

- **Given** an authenticated user **When** they navigate to `/settings/profile` **Then** their current display name, bio, and avatar URL are shown in editable form fields.
- **Given** an authenticated user **When** they submit valid profile changes **Then** the changes are persisted via `PATCH /api/v1/accounts/me` and a success confirmation is shown.
- **Given** an authenticated user **When** they navigate to `/settings/preferences` **Then** their current notification preferences are shown with toggle switches.
- **Given** an authenticated user **When** they toggle a preference and save **Then** the changes are persisted via `PATCH /api/v1/accounts/me/preferences` and a success confirmation is shown.

### US-008: Retrieve Account Details

**As a** frontend client
**I want to** fetch the full account details for the authenticated user
**So that** I can display profile, roles, onboarding status, and preferences.

**Acceptance Criteria:**

- **Given** an authenticated user **When** `GET /api/v1/accounts/me` is called **Then** the response includes `id`, `email`, `display_name`, `bio`, `avatar_url`, `status`, `roles`, `onboarding_completed`, `onboarding_step`, and `notification_preferences`.
- **Given** an unauthenticated request **When** `GET /api/v1/accounts/me` is called **Then** 401 is returned with error code `UNAUTHENTICATED`.

```typescript
export interface NotificationPreferences {

  readonly milestone_completions: boolean;
```

**Updated `create()` method:**

- Sets `bio: null`, `avatarUrl: null`, `onboardingStep: 'welcome'`, `notificationPreferences: DEFAULT_NOTIFICATION_PREFERENCES`.

**New business methods (immutable — all return new Account instances):**

- `withProfile(displayName: string | null, bio: string | null, avatarUrl: string | null): Account`
  - Validates: `displayName` if provided must be 1-100 chars after trim (whitespace-only rejected). `bio` if provided must be 0-500 chars after trim. `avatarUrl` if provided must start with `https://`.
  - Throws: `InvalidAccountDataError` with code `INVALID_ACCOUNT_DATA` for validation failures.
  - Returns: new Account with updated profile fields and `updatedAt = new Date()`.

- `withRoles(roles: readonly AccountRole[]): Account`
  - Validates: `roles` must be a non-empty array. Every element must be a valid `AccountRole`. `'backer'` must always be present.
  - Throws: `InvalidAccountDataError` if validation fails.
  - Returns: new Account with updated roles.

- `withOnboardingStep(step: OnboardingStep): Account`
  - Validates: `step` must be a valid `OnboardingStep`. Step can only advance forward in the sequence: `welcome` -> `role_selection` -> `profile` -> `preferences` -> `completed`. Attempting to regress throws an error.
  - Throws: `InvalidOnboardingStepError` (new error, code `INVALID_ONBOARDING_STEP`) if step is invalid or regression is attempted.
  - Returns: new Account with updated onboarding step. If step is `'completed'`, also sets `onboardingCompleted = true`.

- `withNotificationPreferences(prefs: NotificationPreferences): Account`
  - Validates: all keys must be present. All values must be boolean. `security_alerts` is forced to `true` regardless of input.
  - Throws: `InvalidAccountDataError` if validation fails.
  - Returns: new Account with updated preferences.

- `withDisplayNameFromWebhook(displayName: string | null): Account`
  - Only updates display name if the current `displayName` is `null`. If already set (user has configured it via MMF), the webhook value is ignored.
  - Returns: new Account (possibly unchanged).

## Application Service

### AccountAppService (Modified)

**File:** `packages/backend/src/account/application/account-app-service.ts`

**New dependency (injected):**


**New methods:**

#### `getAccountById(accountId: string): Promise<Account | null>`

1. Call `accountRepository.findById(accountId)`.
2. Return the result.

#### `updateProfile(accountId: string, input: UpdateProfileInput): Promise<Account>`

1. Fetch account via `accountRepository.findById(accountId)`.
2. If not found, throw `AccountNotFoundError` (new error, code `ACCOUNT_NOT_FOUND`).
3. Call `account.withProfile(input.displayName, input.bio, input.avatarUrl)` (domain validation runs here).
4. Persist via `accountRepository.update(updatedAccount)`.
5. Emit `account.profile_updated` event via `eventStore.append(...)`.
6. Log: `{ accountId, action: 'profile_updated' }` (no PII).
7. Return updated account.

**Input type:**

```typescript
export interface UpdateProfileInput {
  readonly displayName?: string | null;
  readonly bio?: string | null;
  readonly avatarUrl?: string | null;
}
```

#### `updateRoles(accountId: string, roles: readonly AccountRole[]): Promise<Account>`

1. Fetch account via `accountRepository.findById(accountId)`.
2. If not found, throw `AccountNotFoundError`.
3. Call `account.withRoles(roles)`.
4. Persist via `accountRepository.update(updatedAccount)`.
5. Emit `account.roles_updated` event.
6. Log: `{ accountId, action: 'roles_updated', roles }`.
7. Return updated account.

#### `advanceOnboardingStep(accountId: string, step: OnboardingStep): Promise<Account>`

1. Fetch account via `accountRepository.findById(accountId)`.
2. If not found, throw `AccountNotFoundError`.
3. Call `account.withOnboardingStep(step)`.
4. Persist via `accountRepository.update(updatedAccount)`.
5. Emit `account.onboarding_step_completed` event with `{ step }`.
6. If `step === 'completed'`, also emit `account.onboarding_completed` event.
7. Log: `{ accountId, action: 'onboarding_step_advanced', step }`.
8. Return updated account.

#### `updateNotificationPreferences(accountId: string, prefs: NotificationPreferences): Promise<Account>`

1. Fetch account via `accountRepository.findById(accountId)`.
2. If not found, throw `AccountNotFoundError`.
3. Call `account.withNotificationPreferences(prefs)` (domain enforces `security_alerts: true`).
4. Persist via `accountRepository.update(updatedAccount)`.
5. Emit `account.preferences_updated` event.
6. Log: `{ accountId, action: 'preferences_updated' }`.
7. Return updated account.

**Modified method:**

#### `handleWebhookEvent` — Clerk display name conflict

In the `user.created` and `user.updated` cases, change `upsertFromWebhook` SQL so that `display_name` is only set if the existing value is NULL:

```sql
ON CONFLICT (clerk_user_id) DO UPDATE SET
  email = EXCLUDED.email,
  display_name = COALESCE(accounts.display_name, EXCLUDED.display_name),
  updated_at = NOW()
```

This ensures that once a user sets their display name via MMF (onboarding or profile edit), the Clerk webhook will not overwrite it.

## API Endpoints

All endpoints below are prefixed with `/api/v1/` following the established pattern.
All endpoints except `/health` require Clerk JWT authentication.
The `user_id` (account UUID) comes from `req.authContext.userId` — never from the request body.

### `GET /api/v1/accounts/me`

**Description:** Returns the full account record for the authenticated user.
**Auth:** Required (Clerk JWT).
**Router file:** `packages/backend/src/account/api/account-router.ts` (new file).

**Success response:** `200 OK`

```json
{
  "data": {
    "id": "uuid",
    "email": "user@example.com",
    "display_name": "Jane Doe",
    "bio": "Mars enthusiast",
    "avatar_url": "https://example.com/avatar.jpg",
    "status": "active",
    "roles": ["backer", "creator"],
    "onboarding_completed": true,
    "onboarding_step": "completed",
    "notification_preferences": {
      "campaign_updates": true,
      "milestone_completions": true,
      "contribution_confirmations": true,
      "new_campaign_recommendations": true,
      "security_alerts": true,
      "platform_announcements": false
    }
  }
}
```

**Error responses:**

| Status | Code | When |
|--------|------|------|
| 401 | UNAUTHENTICATED | No valid Clerk JWT |
| 404 | ACCOUNT_NOT_FOUND | Account not found (should not happen due to JIT creation, but defensive) |

**Relationship to `GET /api/v1/auth/me`:** The existing `/api/v1/auth/me` endpoint remains unchanged.
It is a lightweight endpoint that returns data from `AuthContext` (populated by middleware).
`/api/v1/accounts/me` returns the full account record including new fields (`bio`, `avatar_url`, `onboarding_step`, `notification_preferences`) by querying the database via the app service.
Both endpoints coexist.

**Auth:** Required (Clerk JWT).
**Router file:** `packages/backend/src/account/api/account-router.ts`.

**Zod validation schema:**

```typescript
const updateProfileSchema = z.object({
```

### `PATCH /api/v1/accounts/me/onboarding`

**Description:** Advances onboarding state: updates the onboarding step and optionally updates roles (for role selection step).
**Auth:** Required (Clerk JWT).
**Router file:** `packages/backend/src/account/api/account-router.ts`.

**Request body:**

```json
{
  "step": "role_selection | profile | preferences | completed",
  "roles": ["backer", "creator"]
}
```

**Zod validation schema:**

```typescript
const advanceOnboardingSchema = z.object({
  step: z.enum(['role_selection', 'profile', 'preferences', 'completed']),
  roles: z.array(z.enum(['backer', 'creator'])).min(1).optional(),
}).strict();
```

- `step` is required.
The value represents the step to advance TO (not the step being completed).
`'welcome'` is not a valid target — you can only advance away from it.
- `roles` is optional.
If provided (during the role selection step), the account's roles are updated before the step advances.
Only `'backer'` and `'creator'` are valid role values during onboarding (reviewer/admin/super_admin are assigned by admins only).

**Processing logic:**

1. If `roles` is provided, call `accountAppService.updateRoles(accountId, roles)` first.
2. Then call `accountAppService.advanceOnboardingStep(accountId, step)`.
3. If `step === 'completed'`, the domain automatically sets `onboarding_completed = true`.

**Success response:** `200 OK`

```json
{
  "data": {
    "id": "uuid",
    "onboarding_completed": true,
    "onboarding_step": "completed",
    "roles": ["backer", "creator"]
  }
}
```

**Error responses:**

| Status | Code | When |
|--------|------|------|
| 400 | VALIDATION_ERROR | Request body fails Zod validation |
| 400 | INVALID_ONBOARDING_STEP | Step regression or invalid transition |
| 400 | INVALID_ACCOUNT_DATA | Invalid role values |
| 401 | UNAUTHENTICATED | No valid Clerk JWT |
| 404 | ACCOUNT_NOT_FOUND | Account not found |

| 14 | Admin assigns role during onboarding | Onboarding role selection uses `withRoles` which sets the full roles array. The endpoint should merge with existing non-onboarding roles (reviewer, administrator, super_administrator) rather than replace them. The `updateRoles` app service method should preserve admin-assigned roles when called from the onboarding endpoint. | Integration |
| 15 | Clerk session expires during onboarding | Next API request returns 401. Frontend redirects to sign-in via `ProtectedRoute`. After re-authentication, user is redirected to `/onboarding` and resumes from saved step. | E2E |

| 17 | Event store write failure | Event append and account update should be in the same database transaction. If the event insert fails, the account update is rolled back. The adapter's `update` method and event append should share a transaction. | Integration |

| 28 | PATCH body with extra unknown keys | Zod `.strict()` rejects with `VALIDATION_ERROR`. | Unit |

**Edge case #14 — Role merge strategy for onboarding:**

The onboarding endpoint handler must implement this logic:

1. Fetch current account roles.
2. Separate into onboarding roles (`backer`, `creator`) and admin-assigned roles (`reviewer`, `administrator`, `super_administrator`).
3. Replace only the onboarding roles with the submitted values.
4. Preserve any admin-assigned roles.
5. Pass the merged array to `accountAppService.updateRoles`.

**Edge case #24 — Idempotent completion:**

The `advanceOnboardingStep` handler should catch `InvalidOnboardingStepError` when the account is already at the requested step (or past it) and return 200 with the current account state instead of 400.

## Open Questions

None.
All decisions resolved during research:

1. **Avatar upload vs. URL input** — Deferred to URL input for the local demo. File upload (multipart, magic byte validation, storage) is out of scope. The `avatar_url` column stores a user-provided HTTPS URL.
2. **`/auth/me` vs. `/accounts/me`** — Both coexist. `/auth/me` is the lightweight middleware-populated endpoint. `/accounts/me` is the full account endpoint with all fields.
3. **Clerk display name conflict** — Webhook uses `COALESCE(accounts.display_name, EXCLUDED.display_name)` to avoid overwriting MMF-set names.
4. **Onboarding step regression** — Server enforces forward-only. Backward navigation is client-side state only.
5. **Event store transaction** — Account update and event append in the same database transaction.
6. **GDPR notification defaults** — For the local demo, all categories default to opt-in except platform announcements. Flagged as a production concern for GDPR compliance review.
