> Component-level UI specification. Generated by Design Speccer.

## Overview

feat-004 replaces the onboarding placeholder with a multi-step wizard (welcome, role selection, profile, preferences, completion) and introduces post-onboarding settings pages for profile and notification preferences. The design must feel like a guided journey — the user is being prepared for a Mars mission, not filling out a boring form.

Key design decisions:
- The wizard uses a single-column centred layout (max-width 600px), consistent with the onboarding placeholder established in feat-003
- A horizontal step progress indicator sits below the page heading, showing numbered steps with names
- Role selection uses card-style selection with visual selection state (not radio buttons)
- Toggle switches for notification preferences use a custom component styled to the brand
- The completion step uses `--gradient-celebration` for a success moment
- All pages live inside `PageShell` (header with UserButton, footer)
- Settings pages use the same form components as onboarding for consistency

## Components

### Component: OnboardingWizard

**File:** `packages/frontend/src/pages/onboarding.tsx`
**Reuses:** PageShell, AuthLoadingScreen (from feat-003), ProtectedRoute

This is the page-level orchestrator that manages wizard state and renders the current step.

#### Props

None (page component).

#### Visual Specification

**Outer container:**
- `display: flex`
- `flex-direction: column`
- `align-items: center`
- `flex: 1`
- `padding: 48px 0` (desktop), `32px 0` (tablet), `24px 0` (mobile)

**Content container:**
- `max-width: 600px`
- `width: 100%`
- `padding: 0 16px` (mobile only, handled by PageShell main inner)
- `text-align: center` (for headings and progress indicator)

#### States

**Loading (account query pending):** Render `AuthLoadingScreen` pattern (spinner + "Preparing your mission...").

**Error (account query failed):** Centred error message: "We couldn't load your account. Try refreshing." in `--color-text-error`, with a secondary "Try Again" button.

**Already completed:** Redirect to `/dashboard` (no visual output).

**Default:** Render the step component for `account.onboarding_step`.

#### Accessibility

- **Landmark:** Content is within PageShell's `<main>`.
- **Live region:** Step transitions should use `aria-live="polite"` on the step content container so screen readers announce new step content.
- **Focus management:** When a step transitions, focus should move to the step heading (`<h1>`) so keyboard users know the context has changed.

- `margin-bottom: 32px`

**Step dot (completed):**

- Background: `--color-action-primary`

- Size: 28px diameter
- Background: `--color-action-primary`
- Border: `2px solid var(--color-action-primary)`
- Border-radius: 50%

- Background: `--color-bg-input`
- Border: `1px solid var(--color-border-input)`
- Border-radius: 50%
- Inner content: step number in `--color-text-tertiary`, `--type-button` at 12px

- Text-align: center

**Default:** Dots and labels rendered per current step.

#### Accessibility

- **Role:** `role="navigation"` with `aria-label="Onboarding progress"`

## Responsive Behaviour

### Breakpoints

| Breakpoint | Width | Layout Changes |
|------------|-------|---------------|
| Desktop | >=1024px | Centred 600px max-width content, generous padding (48px top). Role cards in a row. Step labels visible. |
| Tablet | 768-1023px | Centred 600px max-width, reduced padding (32px top). Role cards in a row. Step labels visible. |
| Mobile | <768px | Full-width with 16px horizontal padding, minimal padding (24px top). Role cards stack vertically. Step labels hidden, replaced by "Step X of 5" text. |

### Component-specific responsive notes

- **StepProgressIndicator:** Desktop/tablet shows numbered dots with labels. Mobile shows numbered dots only with a text label for the current step below the dots. Connector lines reduce from 32px to 16px.
- **Role cards:** Stack vertically on mobile (<640px). Side-by-side on sm+ (>=640px). Equal width via `flex: 1`.
- **Page title headings:** Scale from `--type-section-heading` size (40px) on mobile to `--type-page-title` size (56px) on sm+ (>=640px).
- **Form fields:** Full width at all breakpoints. Textarea min-height reduces to 100px on mobile.
- **Button rows:** On mobile (<640px), Back button and right-side buttons may wrap to two lines if horizontal space is tight. Use `flex-wrap: wrap` with `gap: 12px`.
- **Toggle switch rows:** Full width at all breakpoints. Description text wraps naturally. Toggle stays right-aligned.

### Touch Targets

- Role cards: entire card is the touch target (well above 44x44px minimum).
- Toggle switches: 44px wide track meets minimum touch target. Add 24px padding around the toggle for easier tap.
- All buttons: 12px vertical padding + 14px font ensures >= 44px minimum touch target.
- Form inputs: 14px top/bottom padding + 16px font ensures >= 44px minimum touch target.
- Step dots: 28px diameter is below 44px minimum. Add 8px transparent padding around each dot for a 44px minimum hit area.

### Transitions

| Element | Trigger | Animation | Duration | Easing |
|---------|---------|-----------|----------|--------|
| Role card hover | mouseenter | Background to `--color-bg-elevated` | `--motion-hover-duration` | `--motion-hover-easing` |

| KYC callout exit | selection of Backer | Fade out + slide up | `--motion-hover-duration` | `--motion-hover-easing` |
| Step transition | step change | Content fade out then fade in | `--motion-enter-duration` | `--motion-enter-easing` |
| Completion icon | mount | Scale 0.8 to 1.0 + fade in | `--motion-enter-emphasis-duration` | `--motion-enter-emphasis-easing` |
| Success message entry | save success | Slide down + fade in | `--motion-enter-duration` | `--motion-enter-easing` |
| Success message exit | auto-dismiss | Fade out | `--motion-hover-duration` | `--motion-hover-easing` |
| Button hover | mouseenter | Background shift | `--motion-hover-duration` | `--motion-hover-easing` |
| Button loading spinner | mutation in flight | Rotate 360deg continuous | 800ms | linear, infinite |

### Reduced Motion

| Element | Normal Behaviour | Reduced Motion |
|---------|-----------------|----------------|
| Role card hover | Smooth background transition | Instant change (0ms) |
| Toggle switch | Smooth thumb slide | Instant position change |
| KYC callout | Slide + fade | Instant appearance/disappearance |
| Step transition | Fade out/in | Instant switch |
| Completion icon | Scale + fade entrance | Instant appearance |
| Success message | Slide + fade | Instant appearance/disappearance |
| Button loading spinner | Continuous rotation | Static ring at 50% opacity |

**Rule:** No decorative animation. Motion is used only for state transitions (selection, toggle, step change), loading feedback (spinners), and component entry/exit (callout, success message). All animations must honour `prefers-reduced-motion: reduce` per the token system's reduced-motion overrides in `tokens.css`.

## New Patterns Introduced

### 1. Multi-Step Wizard Pattern

A centred single-column wizard with numbered step progress indicator, step content area, and navigation buttons (Back/Skip/Continue). This pattern can be reused for any future multi-step flows (e.g., campaign creation wizard, KYC verification flow).

Key conventions:
- Step progress indicator at the top with numbered dots and connector lines
- Forward navigation persists to server; backward navigation is client-side only
- `aria-live="polite"` on step content for screen reader announcements
- Focus moves to step heading on transition

### 2. Toggle Switch Pattern

A custom toggle switch for boolean on/off states, styled to the MMF brand. This shared component (in `components/ui/`) can be reused for any future toggle needs. Uses `role="switch"` for accessibility.

### 3. Selectable Card Pattern

Cards that function as radio-group items for exclusive selection. Visual selection state via border colour + box-shadow. Uses `role="radiogroup"` and `role="radio"` for accessibility. This pattern can be reused for plan selection, tier selection, or any card-based choice UI.

### 4. Success Notification Banner

An inline success message with auto-dismiss. Uses success status tokens. Announced to screen readers via `role="status"`. This pattern can be reused on any settings page or form submission.

### 5. Settings Page Pattern

A single-column centred settings page with section label, heading, form content, and save button. Consistent layout for all future settings pages (e.g., security settings, payment settings).

## Accessibility Checklist

- [x] All interactive elements are keyboard accessible (buttons are native `<button>`, role cards use radiogroup pattern with arrow key navigation, toggles use `role="switch"` with Enter/Space)
- [x] All images/icons have alt text or aria-label (completion checkmark is `aria-hidden="true"` as decorative; role card icons are decorative with `aria-hidden="true"`; lock icon is decorative)
- [x] Colour is never the sole indicator — always paired with text (selected card has border + "selected" aria state; toggle has position + `aria-checked`; errors have colour + error text; success has colour + message text)
- [x] Focus states are visible on all interactive elements (buttons: `outline: 2px solid var(--color-action-primary-hover); outline-offset: 2px`; inputs: `border-color + box-shadow`; cards: `border-color + box-shadow`; toggles: `outline + outline-offset`)
- [x] Screen reader announces all dynamic content changes (step transitions via `aria-live="polite"`; KYC callout via `role="status"`; success messages via `role="status"`; toggle state changes via `aria-checked`)
- [x] Contrast ratio >= 4.5:1 for all text (WCAG AA) — verified:
