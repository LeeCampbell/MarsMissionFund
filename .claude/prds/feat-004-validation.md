## Warnings (Should Fix)

1. **WARN — Dependency description swap (Spec Writer):** In the Dependencies section, feat-002 is described as "Authentication -- Clerk integration" and feat-003 as "Project scaffold". These descriptions are reversed. feat-002 is Database Schema Foundation and feat-003 is Authentication with Clerk. The dependency IDs are correct. **Recommendation:** Spec Writer should correct the parenthetical descriptions to match the backlog.

2. **WARN — Missing indexes on onboarding columns:** No indexes specified for `onboarding_step` or `onboarding_completed` columns. At local demo scale this is not a problem, but if these columns are used in WHERE clauses (e.g., finding users who need onboarding nudges), indexes would be needed in production. **Recommendation:** Implementation agents should note this; no spec change required for local demo scope.

3. **WARN — Tier 1 token reference in spec text:** The completion step description mentions `--font-display` alongside the correct `--type-page-title`. `--font-display` is a Tier 1 identity token that should not be referenced in component code. The spec text is informational (explaining what `--type-page-title` maps to), but implementation agents should only use `--type-page-title`. **Recommendation:** Spec Writer should remove the `--font-display` reference from the completion step description to avoid confusion.

4. **WARN — No explicit data isolation test case:** The Testing Requirements section does not include an explicit test verifying that `GET /api/v1/accounts/me` cannot return another user's account data. The architecture enforces this via `req.authContext.userId`, but an explicit integration test would strengthen confidence. **Recommendation:** Implementation agents should add a test: "Given user A is authenticated, when GET /accounts/me is called, then only user A's data is returned (not user B's)."

5. **WARN — Transaction boundary not explicitly specified for event emission:** The spec states in edge case #17 and Open Questions #5 that account update and event append should be in the same transaction, but the application service step-by-step logic (steps 4-5 in each method) lists them as sequential steps without explicitly stating they share a database transaction. **Recommendation:** Implementation agents should wrap repository `update` and event store `append` in a shared transaction. The spec should ideally make this explicit in the application service orchestration, but the edge case and open questions sections make the intent clear enough.

6. **WARN — Design spec pending:** The design spec is being written in parallel and does not yet exist. Design system compliance, component specifications, responsive behaviour, accessibility, and animation cannot be fully validated. **Recommendation:** A follow-up validation pass should be performed when the design spec is complete. Implementation of backend and domain layers can proceed without it; frontend implementation should wait for the design spec.
