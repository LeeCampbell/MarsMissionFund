# Research: feat-004 — Account Registration & Onboarding

> Research document for Spec Writer consumption. Generated by Spec Researcher.

## 1. Domain Knowledge

### Key Concepts

- **Onboarding** is a phase within the account lifecycle (not a separate domain concern, per L4-001 Section 2). It is the first-run experience after registration and email verification.
- **Role selection** determines the user's primary intent: Backer (default for all), Creator (additive, requires KYC later), or both. This is not a permanent lock — roles can be changed later.
- **Progressive profiling** means collecting optional profile data (display name, avatar, bio) during onboarding without blocking completion. Users can skip and return later.
- **Onboarding completion** is defined as: the user has selected a role and reached the home surface. Profile completion and KYC are explicitly NOT gates (L4-001 Section 2.2).
- **JIT (Just-In-Time) account creation** is the primary account creation path. On the first authenticated API request, if no `accounts` row exists for the Clerk user ID, one is created with status `active` and default role `backer`. This already works in the codebase via `enrichAuthContext` middleware.
- **Lazy creation vs. webhook creation**: Both paths exist. The `enrichAuthContext` middleware calls `findOrCreateAccount` (JIT path), and the webhook handler processes `user.created` events from Clerk (production reliability path). Both use upsert patterns to handle race conditions.

### Calculation Methodology

No financial calculations for this feature. The only data transformations are:
- Role array management: adding `creator` to the existing `['backer']` array when Creator role is selected.
- Onboarding step tracking: tracking which step the user reached so they can resume.

### Industry Conventions

- **Multi-step onboarding** is standard in crowdfunding platforms. Kickstarter collects basic profile info, location, and interests. Indiegogo differentiates between "contributor" and "campaign owner" during setup.
- **Role selection at onboarding** is common in two-sided marketplaces. The pattern is: default to the more common role (backer/buyer), let users opt into the provider role (creator/seller), and make it easy to add roles later.
- **Progressive profiling** (asking for data gradually rather than upfront) is an established UX pattern that reduces onboarding abandonment. The key principle is: collect only what is needed at each stage, defer the rest.
- **Notification opt-in defaults** vary by jurisdiction. Under GDPR, marketing notifications should default to opt-out. Per L4-001 Section 2.1, MMF defaults all categories to opt-in except platform announcements. Security alerts are always on and cannot be disabled.

### Regulatory Notes

- **GDPR notification consent**: All notification categories that qualify as marketing (new campaign recommendations, platform announcements) should arguably default to opt-out under GDPR. The spec says all categories default to opt-in except platform announcements. For the local demo this is acceptable; flag as a production concern.
- **No PII in logs**: Per L2-002 Section 1.7, account mutations are logged but PII (email, display name) must not appear in log payloads. Log the account ID, event type, and action — not the data itself.
- **Avatar uploads**: File type validation (allowlist: JPEG, PNG, WebP), size limits (e.g. 2MB), and content validation (magic bytes, not just extension) per L2-002 Section 1.4. For the local demo, validation is real but storage can be local filesystem or in-memory.

## 2. Codebase Integration

### Current Data Model

The `accounts` table (migration `20260304000003_create_accounts.sql`) already has all the columns needed for basic onboarding:

```sql
CREATE TABLE accounts (
  id                    UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  clerk_user_id         TEXT         UNIQUE NOT NULL,
  email                 TEXT         UNIQUE NOT NULL,
  display_name          TEXT,
  status                TEXT         NOT NULL DEFAULT 'pending_verification',
  roles                 TEXT[]       NOT NULL DEFAULT '{backer}',
  onboarding_completed  BOOLEAN      NOT NULL DEFAULT FALSE,
  created_at            TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
  updated_at            TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
```

**What is missing from the schema for feat-004:**

1. **`bio`** column — the feature brief and L4-001 Section 4.1 specify a bio field (free-text, sanitised). Needs a new migration to add `bio TEXT` to the accounts table.
2. **`avatar_url`** column — L4-001 Section 4.1 specifies avatar upload. Needs `avatar_url TEXT` column. (Avatar file storage is a separate concern — the column stores the URL/path.)
3. **`onboarding_step`** column — the feature brief says "users who abandon mid-flow resume where they left off". This requires tracking the current/completed onboarding step. Options:
   - A `TEXT` column like `onboarding_step` with values like `welcome`, `role_selection`, `profile`, `preferences`, `completed`.
   - Or an `INTEGER` column like `onboarding_step` (0-based step index).
   - The text approach is more readable and resilient to step reordering.
4. **`notification_preferences`** — the feature brief specifies toggleable notification categories. Options:
   - A new `notification_preferences` table (normalised, one row per category per account).
   - A `JSONB` column on the accounts table (simpler, fewer queries, but less queryable).
   - Given the small fixed set of categories, a `JSONB` column is pragmatic for the local demo.

**Proposed new migration** (`20260304000009_add_onboarding_fields.sql` or similar timestamp):
```sql
-- migrate:up
BEGIN;

ALTER TABLE accounts ADD COLUMN bio TEXT;
ALTER TABLE accounts ADD COLUMN avatar_url TEXT;
ALTER TABLE accounts ADD COLUMN onboarding_step TEXT NOT NULL DEFAULT 'welcome';
ALTER TABLE accounts ADD COLUMN notification_preferences JSONB NOT NULL DEFAULT '{
  "campaign_updates": true,
  "milestone_completions": true,
  "contribution_confirmations": true,
  "new_campaign_recommendations": true,
  "security_alerts": true,
  "platform_announcements": false
}';

ALTER TABLE accounts ADD CONSTRAINT chk_accounts_onboarding_step CHECK (
  onboarding_step IN ('welcome', 'role_selection', 'profile', 'preferences', 'completed')
);

COMMIT;

-- migrate:down
BEGIN;

ALTER TABLE accounts DROP CONSTRAINT IF EXISTS chk_accounts_onboarding_step;
ALTER TABLE accounts DROP COLUMN IF EXISTS notification_preferences;
ALTER TABLE accounts DROP COLUMN IF EXISTS onboarding_step;
ALTER TABLE accounts DROP COLUMN IF EXISTS avatar_url;
ALTER TABLE accounts DROP COLUMN IF EXISTS bio;

COMMIT;
```

### Existing Domain Entities

**`Account` entity** (`packages/backend/src/account/domain/account.ts`):
- Private constructor pattern with `create()` and `reconstitute()` static methods.
- Properties: `id`, `clerkUserId`, `email`, `displayName`, `status`, `roles`, `onboardingCompleted`, `createdAt`, `updatedAt`.
- **Missing**: `bio`, `avatarUrl`, `onboardingStep`, `notificationPreferences`.
- **Missing domain methods**: The entity has no mutation methods. For feat-004, the entity needs methods like:
  - `updateProfile(displayName, bio)` — validates and returns a new Account with updated fields.
  - `updateRoles(roles)` — validates role values and returns updated Account.
  - `completeOnboardingStep(step)` — advances the onboarding step.
  - `completeOnboarding()` — sets `onboardingCompleted = true`.
  - `updateNotificationPreferences(prefs)` — validates and updates preferences.
- The entity is currently immutable (all props readonly, no setters). Mutations should follow the pattern of returning a new Account instance (or the entity needs a controlled internal mutation mechanism).

**`DomainError`** (`packages/backend/src/shared/domain/errors.ts`):
- Base error class with `code` property. `InvalidAccountDataError` already extends it.

### Existing API Endpoints

Current routes:
- `GET /api/v1/auth/me` — returns the authenticated user's account data (auth-router.ts). Already returns `id`, `email`, `display_name`, `status`, `roles`, `onboarding_completed`.
- `POST /api/webhooks/clerk` — processes Clerk webhook events (webhook-router.ts).
- `GET /health` — health check (unauthenticated).

**What feat-004 needs to add:**
- `GET /v1/accounts/me` — returns full account with roles, profile, onboarding status, notification preferences. (Note: the existing `GET /api/v1/auth/me` partially overlaps. Decision needed: extend existing endpoint or add a new one. The feature brief explicitly specifies `GET /v1/accounts/me`.)
- `PATCH /v1/accounts/me` — updates profile fields (display_name, bio, avatar). Zod validation on request body.
- `PATCH /v1/accounts/me/preferences` — updates notification preferences. Zod validation.
- Onboarding-specific endpoints (or the PATCH endpoint handles onboarding mutations too):
  - `PATCH /v1/accounts/me/onboarding` — updates onboarding step and role selection.

**API pattern observations:**
- Routes are prefixed with `/api/v1/` (see auth-router.ts). The feature brief says `/v1/accounts/me` — likely means `/api/v1/accounts/me` following the established pattern.
- Response format: `{ data: { ... } }` for success, `{ error: { code, message, correlation_id } }` for errors.
- Authentication: All endpoints (except `/health` and webhooks) use `requireAuthentication` + `enrichAuthContext` middleware chain.

### Integration Points

1. **Auth middleware chain** (`enrichAuthContext`): Already creates accounts via JIT and attaches `AuthContext` to the request. The `AuthContext` type needs to be extended with new fields (`bio`, `avatarUrl`, `onboardingStep`, `notificationPreferences`).

2. **Dashboard onboarding redirect**: `DashboardPlaceholder` already queries `GET /api/v1/auth/me` and redirects to `/onboarding` if `onboarding_completed` is false. This is the entry point to the onboarding flow.

3. **Event store**: The `events` table exists. Feat-004 acceptance criteria say "All account mutations emit events to the event store." This means every profile update, role change, onboarding step completion, and preference update needs to insert a row into the `events` table. Currently, no code writes to the events table — feat-004 would be the first feature to implement event emission for the account context.

4. **Clerk user data sync**: The `enrichAuthContext` middleware extracts `email` and `displayName` from Clerk claims on every request. If a user changes their display name via the MMF onboarding/profile UI (stored in the `accounts` table), there is a potential conflict with the Clerk-sourced display name. Decision needed: does MMF's display name override Clerk's, or does Clerk remain the source of truth? The current JIT creation uses Clerk's display name as the initial value.

## 3. Third-Party APIs

### Clerk (Authentication)

- **Relevant to feat-004**: Clerk handles email verification, password management, and session management. The onboarding flow runs after Clerk authentication is complete.
- **User metadata**: Clerk supports `publicMetadata` and `privateMetadata` on user objects. MMF could store `onboarding_completed` in Clerk's metadata to avoid a backend round-trip, but the current architecture stores all MMF state in the `accounts` table, which is the correct approach per the hexagonal architecture rules.
- **Avatar**: Clerk provides a `user.imageUrl` (profile image from SSO or uploaded). MMF's avatar is separate from Clerk's — the user could have a Google profile picture in Clerk but set a different avatar in MMF.
- **Email change**: Per the feature brief, "Email change triggers re-verification (handled by Clerk; we display Clerk's UI)." Clerk provides `<UserProfile />` component for this. No custom backend work needed for email changes.
- **Mock strategy**: The existing `MockAuthAdapter` and `MockWebhookVerificationAdapter` in `packages/backend/src/account/adapters/mock/` are sufficient. No additional Clerk mock work needed for feat-004.

### PostHog (Analytics)

- **Relevant to feat-004**: The acceptance criteria mention event emission, but this is domain events to the event store, not PostHog analytics events. However, onboarding completion would be a valuable analytics event for PostHog tracking.
- PostHog integration is not yet implemented in the codebase. This is out of scope for feat-004 but worth noting.
- **Mock strategy**: Feature flags and analytics are stubbed for the local demo.

## 4. Competitor Patterns

### Kickstarter

- **How they handle onboarding**: Minimal onboarding after sign-up. Users land directly on the discover page. Profile completion is available but not prompted. Creator onboarding is a separate flow triggered when starting a project.
- **What works well**: Low friction — users can browse immediately. Creator identity verification happens at project creation time, not at sign-up.
- **What doesn't work**: No explicit role selection means users discover creator features accidentally. New creators often don't understand the approval process until they try to submit.

### Indiegogo

- **How they handle onboarding**: Similar to Kickstarter — minimal onboarding, immediate access to discovery. Creator onboarding is integrated into campaign creation.
- **What works well**: The "I want to raise money" vs "I want to back a project" distinction is surfaced early.
- **What doesn't work**: The campaign creation flow is long and complex, mixing onboarding concerns with campaign setup.

### Patreon

- **How they handle onboarding**: Explicit role selection early: "I'm a creator" vs "I'm a patron". Different onboarding paths based on role. Creator onboarding collects creator-specific info (what they create, payment info, tiers).
- **What works well**: Clear separation of backer and creator paths. Progressive disclosure — only shows relevant features based on role.
- **What doesn't work**: Hard to switch roles later (patron-to-creator transition is not seamless).

### GoFundMe

- **How they handle onboarding**: Very simple — sign up, then either browse or "Start a fundraiser". No formal onboarding wizard.
- **What works well**: Extreme simplicity. Low barrier to entry.
- **What doesn't work**: Lack of profile completion means user profiles are sparse, which reduces trust.

### Key Takeaway for MMF

MMF's approach (explicit multi-step onboarding with role selection) is more structured than most crowdfunding platforms. This is appropriate because:
1. The Creator role has KYC implications (which most competitors also enforce, just later in the flow).
2. MMF's mission-driven nature benefits from knowing user intent early (backer vs creator).
3. The "both" option avoids forcing a premature choice.

The risk is over-engineering onboarding for a workshop demo. Keep steps minimal and skippable.

## 5. Edge Cases

### Data Edge Cases

- [ ] **Empty display name**: User submits an empty string or whitespace-only display name. Expected: reject with validation error. The field is optional, but if provided, must be non-empty after trimming.
- [ ] **Very long display name**: User submits a display name exceeding reasonable limits. Expected: reject with validation error. Suggest a 100-character limit.
- [ ] **Very long bio**: User submits a bio exceeding reasonable limits. Expected: reject with validation error. Suggest a 500-character limit for the local demo.
- [ ] **Display name with special characters**: User enters display name with HTML, SQL, or script injection. Expected: input is sanitised at the boundary (per L2-002 Section 1.4). Store sanitised value.
- [ ] **Bio with HTML/script content**: Same sanitisation concern as display name.
- [ ] **Invalid avatar file type**: User uploads a non-image file (e.g. .exe renamed to .jpg). Expected: validate magic bytes, not just file extension. Reject non-image files.
- [ ] **Oversized avatar file**: User uploads an image exceeding size limit. Expected: reject with clear error message indicating the size limit.
- [ ] **Corrupted avatar file**: File has valid extension and MIME type but corrupted content. Expected: handle gracefully, reject with error.
- [ ] **Null/undefined notification preference values**: PATCH request sends a preference with a non-boolean value. Expected: Zod validation rejects.
- [ ] **Unknown notification category**: PATCH request includes a preference category that doesn't exist. Expected: Zod validation rejects (strict schema).

### Concurrency Edge Cases

- [ ] **Simultaneous profile updates**: User has two tabs open and submits profile updates from both. Expected: last-write-wins is acceptable for profile data. Both writes succeed; final state reflects the last one.
- [ ] **JIT account creation race**: Two simultaneous requests from the same user both trigger `findOrCreateAccount`. Expected: already handled via `ON CONFLICT` upsert in the repository. Second request finds the account created by the first.
- [ ] **Onboarding step update race**: User clicks "next" rapidly, sending multiple step-update requests. Expected: idempotent step updates — setting the same step twice is a no-op. Steps can only advance forward, never backward (or backward is a separate "go back" action).
- [ ] **Role update during onboarding vs. admin role assignment**: An admin assigns a role to a user while the user is in the middle of onboarding. Expected: the admin assignment takes precedence. Onboarding role selection should be additive (add `creator` to existing roles), not replace the entire roles array.

### Integration Edge Cases

- [ ] **Clerk session expires during onboarding**: The user's Clerk session expires mid-onboarding. Expected: the next API request returns 401. The frontend redirects to sign-in. After re-authentication, the user resumes onboarding at their saved step.
- [ ] **API client missing patch method**: The frontend `ApiClient` does not currently have a `patch` method. Calling a PATCH endpoint will fail until this is added. Expected: add `patch` method to `ApiClient`.
- [ ] **Auth context stale after profile update**: After a PATCH to `/v1/accounts/me`, the `AuthContext` attached to the request reflects the pre-update state (because it was enriched before the request handler ran). The response should return the updated data, and the frontend should invalidate the `['auth', 'me']` query cache.
- [ ] **Event store write failure**: If the event insert fails during a profile update, should the profile update be rolled back? Per the CQRS pattern in `domain-knowledge.md`, the event insert and aggregate update should be in the same transaction.
- [ ] **Display name conflict between Clerk and MMF**: Clerk's `user.updated` webhook can overwrite `display_name` in the accounts table. If a user sets a custom display name in MMF, a subsequent Clerk webhook could overwrite it. Expected: once a user sets a display name in MMF, the webhook should not overwrite it (or should only overwrite if the MMF display name is null).

### User Behaviour Edge Cases

- [ ] **User skips all onboarding steps**: User clicks "Skip" on every optional step. Expected: onboarding completes with default values. The user has the default `backer` role, no display name, no bio, no avatar, default notification preferences.
- [ ] **User refreshes page during onboarding**: Expected: the current step is persisted server-side. On page reload, the frontend fetches the account and resumes from the saved step.
- [ ] **User navigates directly to /dashboard without completing onboarding**: Expected: the dashboard checks `onboarding_completed` and redirects back to `/onboarding`. This already works in the current code.
- [ ] **User navigates directly to /onboarding after completing onboarding**: Expected: either redirect to dashboard or show a "you've already completed onboarding" message. The onboarding page should check if onboarding is already complete.
- [ ] **User selects Creator then changes mind**: During onboarding, user selects Creator, sees the KYC message, then wants to go back and select Backer only. Expected: allow backward navigation in the onboarding wizard. Role selection is not committed until the step is confirmed.
- [ ] **User closes browser during onboarding**: Same as "user refreshes page" — step is persisted server-side, resumes on next login.
- [ ] **User signs up via SSO and has no email in Clerk**: Extremely unlikely but possible edge case. Expected: Clerk always provides an email. If somehow missing, the JIT creation will fail (email is NOT NULL). The webhook handler falls back to empty string for email, which would also violate the NOT NULL constraint. This needs defensive handling.

### Boundary Edge Cases

- [ ] **Maximum roles array size**: A user somehow has more roles than the five defined. Expected: domain validation rejects roles not in the allowed set. The Zod schema should enumerate valid roles.
- [ ] **Onboarding step regression**: A PATCH request tries to set `onboarding_step` to a step earlier than the current one. Expected: allow it (user may want to go back) or reject (onboarding is forward-only). The feature brief says "resume where they left off" which implies forward-only tracking, but backward navigation should be handled client-side without persisting the regression.
- [ ] **Notification preferences with extra keys**: PATCH request includes unknown keys in the notification preferences object. Expected: Zod strict schema validation rejects extra keys.
- [ ] **Account in non-active status attempts onboarding**: A suspended or deactivated account is redirected to onboarding. Expected: the `enrichAuthContext` middleware already blocks suspended/deleted accounts before they reach any route handler. This is handled.
- [ ] **Multiple rapid onboarding completions**: User submits the final onboarding step multiple times. Expected: idempotent — setting `onboarding_completed = true` when it is already true is a no-op.

## 6. Recommendations

### Must-Haves for Spec

1. **New migration for missing columns**: The accounts table needs `bio`, `avatar_url`, `onboarding_step`, and `notification_preferences` columns. The spec must include the migration SQL.

2. **Account entity mutation methods**: The `Account` domain entity has no mutation methods. The spec must define how profile updates work: either return new Account instances (functional immutability) or use controlled internal mutation. Given the `readonly` props pattern, returning new instances via methods like `withDisplayName(name)` or `withRoles(roles)` is the cleanest approach.

3. **API client `patch` method**: The frontend API client is missing a `patch` method. The spec must include adding this.

4. **Event emission pattern**: This is the first feature that writes to the event store. The spec must define:
   - Event types (e.g. `account.profile_updated`, `account.roles_updated`, `account.onboarding_completed`, `account.preferences_updated`).
   - Event payload shapes.
   - Transactional pattern (event insert + account update in same transaction).
   - An `EventStore` port and adapter.

5. **Zod validation schemas**: Define schemas for:
   - `PATCH /v1/accounts/me` body (display_name, bio optional strings with length limits).
   - `PATCH /v1/accounts/me/preferences` body (notification preference toggles).
   - Onboarding step update body (role selection, step advancement).

6. **Onboarding step tracking**: Define the exact steps and their order. Proposed: `welcome` -> `role_selection` -> `profile` -> `preferences` -> `completed`. Clarify whether "welcome" is a persisted step or just a UI screen.

7. **`GET /v1/accounts/me` vs `GET /v1/auth/me`**: The existing auth endpoint returns account data. The feature brief specifies a new accounts endpoint. The spec must clarify the relationship: is `/v1/auth/me` deprecated in favour of `/v1/accounts/me`, or do they coexist with different scopes?

8. **Clerk display name conflict resolution**: Define whether the Clerk webhook should overwrite MMF's display name. Recommendation: the webhook should only set `display_name` if the current value in the accounts table is NULL.

### Watch-Outs

1. **Event store is not yet used**: No code currently writes to the `events` table. Feat-004 would be the first consumer. This means implementing the event store port, adapter, and transactional write pattern from scratch. This adds scope.

2. **Avatar upload complexity**: File upload (multipart form data, validation, storage) is significantly more complex than text field updates. For the local demo, consider deferring avatar upload to a later feature or using a simple URL input instead. If implemented, it needs:
   - Multipart form data parsing (e.g. `multer` for Express).
   - File type validation (magic bytes).
   - File size validation.
   - Storage (local filesystem or S3 for prod).
   - URL generation for retrieval.

3. **Notification preferences default mismatch**: L4-001 Section 2.1 says "All categories default to opt-in except platform announcements." But L4-001 Section 4.2 says security alerts "cannot be disabled." Ensure the schema enforces that `security_alerts` is always `true` regardless of what the user sends.

4. **Onboarding step tracking adds state**: The `onboarding_step` column introduces a mini state machine. Ensure transitions are validated at the domain layer, not just the database constraint.

5. **Frontend state management for multi-step form**: A multi-step onboarding wizard needs local state for the current step's form data, plus server-persisted state for completed steps. TanStack Query's `useMutation` with optimistic updates could be used, but the interaction between local form state and server state needs careful design.

### Suggested Approach

1. **Migration first**: Add the missing columns to the accounts table.
2. **Domain entity methods**: Add mutation methods to the `Account` entity for profile updates, role changes, onboarding step advancement, and preference updates.
3. **Repository methods**: Add `update` method to `AccountRepository` that persists changes to the accounts table.
4. **Application service methods**: Add methods to `AccountAppService` for profile update, onboarding step update, role selection, and preference update. Each method should validate input, update the entity, persist, and emit an event.
5. **Event store port and adapter**: Implement a minimal event store port and PG adapter for writing events.
6. **API routes**: Add `GET /api/v1/accounts/me`, `PATCH /api/v1/accounts/me`, `PATCH /api/v1/accounts/me/preferences`, and optionally `PATCH /api/v1/accounts/me/onboarding` with Zod validation.
7. **Frontend API client**: Add `patch` method.
8. **Frontend onboarding page**: Replace the placeholder with a multi-step wizard (welcome, role selection, profile, preferences). Use local React state for the current step and form data, persist completed steps via API.
9. **Frontend profile page**: Add a new route (`/profile` or `/settings/profile`) for post-onboarding profile editing.
10. **Avatar upload**: Defer to a follow-up or implement as a simple URL input for the local demo.
