## Summary

### Medium (Should Fix)

#### MED-001: No CHECK constraint on `accounts.roles` array values

- **Location:** `db/migrations/20260304000003_create_accounts.sql:10`
- **Issue:** The `roles` column is defined as `TEXT[] NOT NULL DEFAULT '{backer}'` but there is no CHECK constraint restricting which values can appear in the array. While the spec defines valid roles (backer, creator, reviewer, administrator, super_administrator), nothing at the database level prevents insertion of arbitrary role strings such as `'{superadmin}'` or `'{root}'`.
- **Recommendation:** Add a CHECK constraint using a function or unnest-based check to validate that all array elements are in the allowed set. Alternatively, document that role validation is strictly an application-layer responsibility. Example:
  ```sql
  CONSTRAINT chk_accounts_roles CHECK (
    roles <@ ARRAY['backer','creator','reviewer','administrator','super_administrator']::TEXT[]
  )
  ```

#### MED-002: `escrow_ledger.contribution_id` FK not enforced

- **Location:** `db/migrations/20260304000007_create_escrow_ledger.sql:9`
- **Issue:** The `contribution_id` column is declared as `UUID` but has no FOREIGN KEY constraint referencing `contributions(id)`. Similarly, `disbursement_id` has no FK constraint. While the spec does define these as nullable columns without explicit FK requirements, the lack of referential integrity means orphaned references could be inserted. The infrastructure rules state "Explicit ON DELETE on every FK" -- these columns lack FKs entirely.
- **Recommendation:** Add FK constraints with appropriate ON DELETE behaviour (likely RESTRICT), or document in the spec that these are intentionally denormalised references for auditability reasons (since the ledger is append-only, CASCADE delete would conflict with immutability).

#### MED-003: No length constraint on TEXT columns

- **Location:** Multiple migration files (accounts, campaigns, milestones, contributions, kyc_verifications, events)
- **Issue:** Several TEXT columns have no length constraints: `accounts.display_name`, `campaigns.title`, `campaigns.description`, `milestones.title`, `milestones.description`, `milestones.verification_criteria`, `events.event_type`, `events.aggregate_type`, `events.source_service`, `escrow_ledger.description`, `kyc_verifications.document_type`, `kyc_verifications.provider_reference`, and `contributions.gateway_reference`. Unbounded TEXT columns can be used for denial-of-service via storage exhaustion. The `campaigns.summary` column correctly uses `VARCHAR(280)`.
- **Recommendation:** Add reasonable length limits via CHECK constraints or VARCHAR types on columns that accept user-influenced input. This is a defence-in-depth measure -- primary input validation should occur at the application layer, but database-level limits provide a safety net. Priority columns: `campaigns.title`, `accounts.display_name`, `milestones.title`.

### Low / Informational

#### LOW-001: Event store `payload` column has no size limit

- **Note:** The `events.payload` column is `JSONB NOT NULL DEFAULT '{}'` with no size constraint. While JSONB storage is efficient, extremely large payloads could impact query performance and storage costs. Consider adding a CHECK constraint such as `CHECK (pg_column_size(payload) < 65536)` or documenting a maximum payload size convention for application-layer enforcement.

#### LOW-002: No row-level security (RLS) policies defined

- **Note:** None of the tables define PostgreSQL Row Level Security policies. Data isolation will rely entirely on application-layer query scoping (WHERE user_id = $1). This is acceptable for the current architecture (backend rules specify "Every query scoped to the authenticated user_id from auth context"), but RLS would provide defence-in-depth at the database layer. This is informational and not required for a workshop sample app, but worth noting for production-grade deployments.

#### LOW-003: `milestones` uses ON DELETE CASCADE while other child tables use RESTRICT

- **Note:** `milestones.campaign_id` references `campaigns(id) ON DELETE CASCADE`, while `contributions.campaign_id` and `escrow_ledger.campaign_id` both use `ON DELETE RESTRICT`. This is likely intentional (milestones are owned by the campaign, while contributions and ledger entries are financial records that should never be silently deleted). The inconsistency is deliberate and correct, but worth documenting for future developers.

#### LOW-004: `accounts.email` stored in plaintext

- **Note:** The `email` column in accounts is stored as plaintext TEXT. For a workshop/sample application this is acceptable. For a production financial application, consider whether email addresses should be encrypted at rest at the application layer or whether database-level encryption (pgcrypto) is warranted, depending on regulatory requirements.
